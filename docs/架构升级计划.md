# ProductScout-A2A 架构升级计划

> 基于 SmartVoyage 对比分析，提取可落地的改进方案。本文档可直接作为实施清单使用。

---

## 总览

| 优先级 | 模块 | 当前状态 | 目标状态 | 工作量 |
|--------|------|----------|----------|--------|
| **P0** | 响应速度优化 | 每次请求 21 次 LLM + 20 次 TCP 握手 | MCP 连接池 + Agent 按需调度 → LLM 调用减半 | 小 |
| **P0** | 记忆系统 | context_json 扁平 JSON | 独立的 ConversationMemory 类 + MySQL 持久化 | 中 |
| **P0** | 意图识别 | 正则关键词匹配 | LLM 主路径 + 正则兜底 | 中 |
| **P1** | 多轮上下文注入 | 仅合并槽位 | history + profile + task_context 三维注入 | 小 |
| **P2** | ReAct Agent | 固定 JSON-in/JSON-out | LangChain Agent 框架 | 大 |
| **P2** | Docker 部署 | 手动启动 9 个进程 | 单一 Dockerfile + docker-compose 一键部署 | 中 |

## Phase 0: 响应速度优化 — P0

> 核心问题：每次选品请求产生 21 次 LLM API 调用 + 20 次 TCP 新建/销毁。
> 目标：LLM 调用减半，MCP 连接只建一次。

### 0.1 MCP 连接池化 — `backend/a2a_base.py`

**现状**：每次 `call_mcp_tool()` 创建新的 `httpx.AsyncClient`，用完销毁。

```python
# 改前 — 每次新建连接
async def call_mcp_tool(mcp_url: str, tool_name: str, params: dict) -> dict:
    async with httpx.AsyncClient(timeout=MCP_TIMEOUT_SECONDS) as client:
        resp = await client.post(url, json=params)
        ...
```

**改造**：全局单例 AsyncClient，复用 TCP 连接。

```python
# 改后 — 全局复用
_mcp_client: httpx.AsyncClient | None = None

def get_mcp_client() -> httpx.AsyncClient:
    global _mcp_client
    if _mcp_client is None or _mcp_client.is_closed:
        _mcp_client = httpx.AsyncClient(timeout=MCP_TIMEOUT_SECONDS,
                                         limits=httpx.Limits(max_keepalive_connections=20))
    return _mcp_client

async def call_mcp_tool(mcp_url: str, tool_name: str, params: dict) -> dict:
    client = get_mcp_client()
    resp = await client.post(url, json=params)
    ...
```

**效果**：消除 20 次 TCP 握手 ≈ 省 1 秒。

### 0.2 Agent 按需调度 — `backend/selection_service.py`

**现状**：无条件对每个候选商品调全部 4 个 Agent。

```python
# 改前 — 5 商品 × 4 Agent = 20 次 LLM
A2A_AGENTS = {
    "MarketAgent": MARKET_A2A_URL,
    "ProfitAgent": PROFIT_A2A_URL,
    "SupplyRiskAgent": SUPPLY_RISK_A2A_URL,
    "ReviewInsightAgent": REVIEW_A2A_URL,
}
```

**改造**：根据用户意图和槽位动态选择 Agent 组合。

```python
# 改后 — 按需选择
def select_agents(user_intent, slots) -> dict:
    agents = {}

    # 市场分析：追趋势/低竞争时必须
    if "trend_chasing" == user_intent.goal or "低竞争" in slots.get("preferences", []):
        agents["MarketAgent"] = MARKET_A2A_URL
        agents["ReviewInsightAgent"] = REVIEW_A2A_URL  # 口碑辅助判断

    # 利润分析：高利润需求时必须
    if "profit_first" == user_intent.goal:
        agents["ProfitAgent"] = PROFIT_A2A_URL

    # 风险分析：规避风险/新手时必须
    if user_intent.goal == "risk_averse" or user_intent.experience_level == "newbie":
        agents["SupplyRiskAgent"] = SUPPLY_RISK_A2A_URL

    # 兜底：至少保留 2 个核心 Agent
    if len(agents) < 2:
        agents["MarketAgent"] = MARKET_A2A_URL
        agents["ProfitAgent"] = PROFIT_A2A_URL

    return agents
```

**效果**：平均从 4 Agent 降至 2-3 Agent → LLM 调用从 21 次降至 6-12 次。

### 0.3 减少候选商品数量

**现状**：`top_k=5` 时分析 5 个商品。

**改造**：先粗筛 Top 3，分析完展示后让用户决定是否查看更多。

```python
# _stream_selection() 中
candidates = query_candidate_products(constraints, top_k * 3)
products_to_analyze = candidates[:min(top_k, 3)]  # 最多 3 个
```

**效果**：非必要情况下 5→3 商品，LLM 调用从 21→13 次。

### 0.4 改造文件清单

| 文件 | 改动 |
|------|------|
| `backend/a2a_base.py` | MCP 客户端全局单例 + 连接池 |
| `backend/selection_service.py` | `select_agents()` 函数，`analyze_single_product` 改用动态 Agent 列表 |
| `backend/session_orchestrator.py` | `_stream_selection()` 中 top_k 上限限制 |

## Phase 1: 记忆系统 (Memory) — P0

### 1.1 目标

将当前 `context_json` 扁平结构替换为 SmartVoyage 风格的四组件记忆系统：

```
ConversationMemory
├── short_term_messages  最近 10 轮对话
├── user_preferences     用户偏好 (如 preferred_categories, price_range)
├── current_task         当前任务上下文 (当前意图/已补齐槽位/追问轮次)
└── query_history        历史查询记录 (最近 20 条)
```

### 1.2 新增文件 `backend/memory.py`

```python
class ConversationMemory:
    """四组件记忆管理器，支持 MySQL 持久化和服务重启恢复"""

    def __init__(self, short_term_limit: int = 10):
        self.short_term_messages = []   # [{"role","content","timestamp"}, ...]
        self.user_preferences = {}      # {"preferred_categories": [...], "price_range": {...}}
        self.current_task = {}          # {"intent","slots","follow_up_count","last_query"}
        self.query_history = []         # [{"intent","query","timestamp"}, ...]
        self.short_term_limit = short_term_limit
        self._db_conn = None

    # ── 读写操作 ──
    def add_message(self, role, content)           # 追加消息，超限自动截断
    def get_short_term_text(self) -> str           # 格式化 "User: ...\nAssistant: ..."
    def update_preferences(self, prefs: dict)       # 合并更新
    def get_preference_text(self) -> str            # 格式化 "类目偏好：家居，价格偏好：100-300"
    def update_task_context(self, ctx: dict)        # 合并更新
    def add_query(self, intent, query)              # 追加历史 Query

    # ── 持久化 ──
    def set_db_connection(self, conn)
    def save_messages_to_db()
    def load_messages_from_db()
    def save_preferences_to_db()
    def load_preferences_from_db()
    def save_query_to_db()
    def load_queries_from_db()

    # ── 序列化 ──
    def to_dict(self) -> dict
    @classmethod
    def from_dict(cls, data) -> ConversationMemory

    # ── 清理 ──
    def clear()
    def clear_all_from_db()
```

### 1.3 新增数据表

```sql
-- 用户偏好 (键值对持久化)
CREATE TABLE IF NOT EXISTS user_preferences (
    id INT PRIMARY KEY AUTO_INCREMENT,
    pref_key VARCHAR(100) NOT NULL,
    pref_value TEXT NOT NULL,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_key (pref_key)
) ENGINE=InnoDB;

-- 查询历史
CREATE TABLE IF NOT EXISTS query_history (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    intent_type VARCHAR(50) NOT NULL,
    query_content TEXT NOT NULL,
    query_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_time (query_time)
) ENGINE=InnoDB;

-- 系统记忆 (持久化 short_term_messages)
CREATE TABLE IF NOT EXISTS system_memory (
    id INT PRIMARY KEY AUTO_INCREMENT,
    role VARCHAR(20) NOT NULL,
    content TEXT NOT NULL,
    message_time VARCHAR(20),
    message_order INT DEFAULT 0
) ENGINE=InnoDB;
```

### 1.4 改造点

- `session_orchestrator.py`: 将 `context_json` 操作替换为 `ConversationMemory` 实例
- `session_service.py`: `SessionManager` 内的 `update_context` / `add_message` 委托给 memory
- ChatService 初始化时 `memory.load_all_from_db()` 恢复数据
- 每次 `add_message` / `update_preferences` 后立即 `save_xxx_to_db()`

## Phase 2: LLM 意图识别 — P0

### 2.1 目标

将当前 `_is_selection_request()` + `_extract_slots()` 的正则匹配，升级为 LLM 主路径 + 正则兜底。

### 2.2 当前 vs 目标

| | 当前 | 目标 |
|------|------|------|
| 意图分类 | `_is_selection_request()` 关键词匹配 | LLM 意图 Prompt → 取 `intent` 字段 |
| 槽位提取 | `_extract_slots()` 正则规则 | LLM 意图 Prompt → 取 `slots` 字段 |
| 查询改写 | 不做 | LLM 结合 conversation_history 自动改写 |
| 追问生成 | `_build_follow_up()` 硬编码模板 | LLM 生成 `follow_up_message`，模板作为 fallback |
| 兜底 | 无 | `MOCK_LLM=true` 或 LLM 失败时走当前正则逻辑 |

### 2.3 新增 Prompt `INTENT_SLOT_RECOGNIZE_V1`

```python
INTENT_SLOT_RECOGNIZE_V1 = """你是电商选品对话助手。分析用户输入，完成意图分类、槽位抽取、查询改写和追问。

支持的意图:
- product_selection: 用户想选品/找产品/上新
- report_lookup: 用户想看之前的分析报告
- product_compare: 用户想对比多个商品
- supply_inquiry: 用户想了解供应链/供应商/风险
- slot_reply: 用户正在回答系统的追问
- general_chat: 问候、能力询问

槽位定义:
- category: 商品类目 (如"家居小电器""厨房用品")
- season: 季节或场景 (如"夏季""冬季""日常")
- price_min/price_max: 价格区间
- preferences: 偏好列表 (如["高利润","低竞争"])
- target_product_ids: 目标商品ID列表

对话历史: {conversation_history}
当前任务上下文: {task_context}
用户偏好: {user_preferences}
用户输入: {query}

输出严格 JSON:
{{
  "intent": "product_selection",
  "confidence": 0.9,
  "rewritten_query": "帮我选家居小电器，夏季上新，价格100-300元，高利润低竞争",
  "slots": {{
    "category": "家居小电器",
    "season": "夏季",
    "price_min": 100,
    "price_max": 300,
    "preferences": ["高利润", "低竞争"]
  }},
  "slots_missing": [],
  "follow_up_message": ""
}}
"""
```

### 2.4 改造点

- `session_orchestrator.py` 新增 `_llm_recognize_intent()` 函数
- 保留 `_extract_slots()` / `_build_follow_up()` / 关键词匹配作为 fallback
- 决策逻辑：`MOCK_LLM=false && API_KEY 存在` → LLM 路径，否则走正则

## Phase 3: 多轮上下文注入 — P1

### 3.1 目标

将 Memory 的三个维度注入 LLM prompt，使 LLM 能理解「聊到哪了」和「用户是谁」。

### 3.2 注入点

LLM 意图识别时注入（对应 Phase 2 的 Prompt 变量）：

```
对话历史: {conversation_history}   ← memory.get_short_term_text()
当前任务: {task_context}           ← json.dumps(memory.current_task)
用户偏好: {user_preferences}       ← memory.get_preference_text()
```

### 3.3 效果

当前：
```
用户: "家居小电器"  → 正则不认识 → 返回 generic_reply ❌
```

目标：
```
memory.current_task = {"intent":"product_selection","slots_missing":["category"]}
memory.short_term_messages = [..., "系统: 请问您想选哪个类目？"]
用户: "家居小电器"
  → LLM 看到 task_context.slots_missing=["category"] → 知道在回答追问
  → LLM 看到 conversation_history → 知道是补类目
  → 输出: {intent: "slot_reply", slots: {category: "家居小电器"}}
```

### 3.4 改造点

- `session_orchestrator.py` 的 `stream_message()` 中，在 LLM 调用前组装三维上下文
- 每次 slot_reply 合并后更新 `memory.current_task`
- 用户偏好自动学习：如果用户多次选「家居小电器 + 100-300」，自动写入 `user_preferences`

## Phase 4: ReAct Agent 框架 — P2

### 4.1 目标

将当前 A2A Agent 的固定「收 Task → 调 MCP → 调 LLM → 返回 JSON」流程，升级为 LangChain ReAct Agent。

### 4.2 当前 vs 目标

**当前** (以 MarketAgent 为例)：
```python
async def handler(input_data, logger):
    product = input_data["product"]
    mcp_data = await call_mcp_tool("get_market_trends", {"category": product["category"]})
    llm_input = format_prompt(product, mcp_data)
    result = call_llm(system_prompt, llm_input)
    return parse_json(result)
```

**目标**：
```python
# Agent 定义时注册 MCP Tools
tools = [
    get_market_trends_tool,
    get_competitor_data_tool,
    get_price_band_tool,
]

agent = create_react_agent(llm, tools, MARKET_AGENT_SYSTEM_V1)
executor = AgentExecutor(agent, tools, verbose=True)

async def handler(input_data, logger):
    query = f"分析商品 {input_data['product']['product_name']} 的市场机会"
    result = executor.invoke({"query": query, "product": input_data["product"]})
    return parse_json(result["output"])
```

### 4.3 优势

- Agent 自主选择调用哪些 MCP Tool（不需要预先决定）
- 支持多步推理（先查趋势，如果趋势分低 → 跳过竞争分析）
- 错误时 Agent 可以重试不同的 Tool 参数

### 4.4 依赖

- 安装 `langchain` + `langchain-openai`
- 每个 A2A Agent 重构为 LangChain Tool 定义 + ReAct Agent
- MCP Tool 注册到 Agent 的 tools 列表

## Phase 5: Docker 一键部署 — P2

### 5.1 目标

一个 `docker-compose up -d` 命令启动全部 9 个服务。

### 5.2 新增文件

**`Dockerfile`** — 单一镜像，所有服务共用：
```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY backend/requirements.txt .
RUN pip install -r requirements.txt
COPY backend/ .
EXPOSE 8090
CMD ["python", "api_server.py"]
```

**`docker-compose.yml`** 改造 — 加入 A2A/MCP 服务定义：

```yaml
services:
  mysql:
    # 已有，不变

  milvus:
    # 已有，不变

  market-a2a:
    build: .
    command: python a2a_server/market_server.py
    ports: ["5101:5101"]
    depends_on: [market-mcp]
    environment:
      - MYSQL_HOST=mysql
      - MARKET_MCP_URL=http://market-mcp:8101/mcp

  market-mcp:
    build: .
    command: python mcp_server/mcp_market_server.py
    ports: ["8101:8101"]
    depends_on: [mysql]

  # ... profit / supplyrisk / review 同理

  api-server:
    build: .
    command: python api_server.py
    ports: ["8090:8090"]
    depends_on: [market-a2a, profit-a2a, supplyrisk-a2a, review-a2a]
```

### 5.3 `config.py` Docker 自动检测

```python
import os

def _is_docker():
    return os.path.exists('/.dockerenv')

_A2A_HOST = "host.docker.internal" if _is_docker() else "localhost"
_MCP_HOST = "host.docker.internal" if _is_docker() else "localhost"
```

服务间 URL 通过 Docker service name 通信（如 `http://market-a2a:5101`），本地开发用 `localhost`。
