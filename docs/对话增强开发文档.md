## 0. 现状与目标

### 0.1 当前架构

```
POST /api/selection/analyze
  └─ run_selection()
       ├─ parse_selection_request()   → 槽位抽取 (LLM)
       ├─ query_candidate_products()  → SQL 查询
       ├─ analyze_single_product() × N
       │    ├─ call_a2a_agent("MarketAgent")
       │    ├─ call_a2a_agent("ProfitAgent")
       │    ├─ call_a2a_agent("SupplyRiskAgent")
       │    └─ call_a2a_agent("ReviewInsightAgent")
       ├─ calculate_final_score()     → 加权评分
       └─ generate_report()           → LLM 生成报告
```

当前是 **单次请求-响应** 模式。前端是一个表单页，输入需求 → 等待 → 看表格和报告。

### 0.2 缺失的能力

| 能力 | 现状 |
|------|------|
| 意图识别 | 无分类，假设所有输入都是选品需求 |
| 槽位识别 | 只做一次抽取，槽位不全也无法追问 |
| 动态规划 | 固定流程，不会根据中间结果调整 |
| 用户意图识别 | 无法判断用户业务场景/角色/深层动机 |
| 历史追问 | 无会话管理，每次请求独立 |

### 0.3 目标架构

```
POST /api/chat/send
  └─ SessionOrchestrator
       ├─ UserIntentRecognizer → 用户意图识别 (业务场景/角色/动机)
       ├─ IntentClassifier   → 意图分类 (选品/查报告/对比/供应链/闲聊)
       ├─ SlotFiller         → 槽位抽取 + 追问补全
       ├─ PlanEngine         → 动态生成执行计划
       ├─ PlanExecutor       → 按计划调度 Agent / 查 DB
       │    ├─ 选品分析 → 复用现有 A2A Agent
       │    ├─ 报告查询 → 直接查 selection_reports 表
       │    ├─ 商品对比 → 调 Agent 做对比分析
       │    └─ 供应链咨询 → 单独调 SupplyRiskAgent
       └─ FollowUpManager    → 缺槽位则追问，结合历史增强
```

## 1. 用户意图识别 (UserIntentRecognizer)

### 1.1 定义

**用户意图识别** 和 **意图识别** 是两个不同的层：

| 层级 | 识别目标 | 示例 |
|------|----------|------|
| **用户意图识别** | 用户是什么角色/场景/深层动机 | "新手卖家想入门"、"老手优化利润率"、"季节性备货"、"清仓甩卖" |
| **意图识别** (2节) | 用户这次想做什么具体操作 | 选品分析、查报告、商品对比、供应链咨询 |

### 1.2 业务价值

同一句话 "帮我选家居"，用户意图不同，回复策略完全不同：

```
用户: 帮我选家居小电器

场景1: 新手试探 → 推荐低门槛、低MOQ、退货率低的入门款
场景2: 老手优化 → 推荐高毛利、差异化强、竞争有壁垒的进阶款
场景3: 季节性备货 → 结合当前季节推荐应季爆款，强调时效
场景4: 清仓甩卖 → 推荐价格带低、走量快的尾货机会
```

### 1.3 识别维度

| 维度 | 值 | 从什么信号判断 |
|------|------|--------------|
| `experience_level` | `newbie` / `intermediate` / `veteran` | 用词专业度、是否提及 MOQ/ROI/供应链等术语 |
| `urgency` | `urgent` / `planned` / `exploring` | 是否含"急""马上""这周"等时间词 |
| `goal` | `profit_first` / `volume_first` / `risk_averse` / `trend_chasing` | "利润高""走量""稳一点""爆款" |
| `scale` | `trial` / `small_batch` / `bulk` | "先试试""小批量""大量" |
| `scenario` | `new_store` / `restock` / `seasonal` / `clearance` | "新店""补货""夏天""清仓" |

### 1.6 对后续流程的影响

```
用户意图 → 影响槽位 → 影响 Plan → 影响 Agent 权重 → 影响报告措辞

具体例子:
  识别到 { experience: "newbie", scale: "trial" }
    → influence_slots: 自动设 profit_requirement="medium, price_max=200"
    → influence_plan: 增加 "入门建议" step, 调高 review_score 权重
    → Agent 权重: 降低 risk_score 权重(新手看不懂), 调高 review_score(口碑直观)
    → 报告措辞: 避免术语, 强调"退货率低""好评多""供应商靠谱"
```

## 2. 意图识别 + 槽位识别

### 2.1 合并为一次 LLM 调用

意图和槽位在同一 prompt 中完成分类 + 抽取，减少 LLM 调用次数：

```
用户输入 → LLM (INTENT_CLASSIFY_AND_SLOT_V1)
         → 返回 { intent, slots, confidence }
```

### 2.2 意图分类定义

| 意图 | intent | 说明 | 后续动作 |
|------|--------|------|----------|
| 选品分析 | `product_selection` | 帮我在某个类目/价位选品 | 槽位补全 → 调 Agent 分析 |
| 查询报告 | `report_lookup` | 查之前的选品报告 | 直接查 DB |
| 商品对比 | `product_compare` | 对比指定商品 | 查商品 → 调 Agent 对比 |
| 供应链咨询 | `supply_inquiry` | 问某类目供应链风险/供应商 | 单调 SupplyRiskAgent |
| 追问回复 | `slot_reply` | 回答系统的追问 (如"类目是家居") | 合并槽位 → 继续 |
| 闲聊/其他 | `general_chat` | 问候、能力询问等 | 直接回复 |

### 2.3 槽位定义

| 槽位 | key | 类型 | 必填 | 追问策略 |
|------|-----|------|------|----------|
| 类目 | `category` | string | 选品/供应链必填 | 候选商品多时追问 |
| 价格下限 | `price_min` | number | 建议 | 不强制追问 |
| 价格上限 | `price_max` | number | 建议 | 不强制追问 |
| 季节/场景 | `season` | string | 建议 | 结合趋势追问 |
| 偏好 | `preferences` | string[] | 建议 | 根据画像推荐 |
| 目标商品ID | `target_product_ids` | int[] | 对比/查报告必填 | 列出候选让用户选 |
| 利润要求 | `profit_requirement` | string | 建议 | high/medium/low |

### 2.6 槽位追问逻辑

```
IntentResult.slots_missing 非空
  → 不执行后续流程
  → FollowUpManager 生成追问消息
  → 追问消息带选项按钮 (如"家居"、"数码"、"运动户外")
  → 用户点击选项或输入后 → slot_reply 意图 → 合并槽位 → 重新检查
```

追问最多 **3 轮**，3 轮后槽位仍不全则用默认值直接执行。


## 4. 动态规划 (PlanEngine)

### 4.1 设计思路

不用 LLM 生成计划（不稳定、不可控），用 **决策表 + 规则** 动态组合执行步骤。

```
PlanEngine 输入: intent + slots + user_intent
           输出: Plan (有序的 Step 列表)

PlanExecutor 执行: for step in plan → 调用 → 收集结果 → 检查是否需调整
```

### 4.2 决策表 (PlanEngine 核心)

| intent | 条件 | Plan Steps |
|--------|------|------------|
| `product_selection` | 槽位齐全 | query_candidates → [MarketAgent, ProfitAgent, SupplyRiskAgent, ReviewInsightAgent] → score → report |
| `product_selection` | 槽位不全且首轮 | generate_follow_up (引导式) |
| `product_selection` | 槽位不全且≥2轮 | quick_search (先用现有槽位粗筛，展示 Top3，让用户确认方向) |
| `product_selection` | 候选商品 < 3 | expand_criteria (放宽条件) → 重新查 → 追问用户确认 |
| `report_lookup` | — | query_reports → format_as_card |
| `product_compare` | 有 product_ids | lookup_products → [MarketAgent, ReviewInsightAgent] → comparison_report |
| `supply_inquiry` | — | SupplyRiskAgent only → supply_report |
| `general_chat` | — | llm_chat_reply |

### 4.3 Plan 数据结构

```python
@dataclass
class PlanStep:
    step_id: str            # "fetch_candidates" | "market_analysis" | ...
    step_type: str          # "query" | "agent_call" | "score" | "report" | "follow_up"
    agent_name: str | None  # MarketAgent / ProfitAgent / ...
    depends_on: list[str]   # 依赖的 step_id
    params: dict
    retry_on_fail: bool

@dataclass
class Plan:
    intent: str
    steps: list[PlanStep]
    fallback: str           # 异常兜底动作
```

```python
    # 3. 意图分类 + 槽位抽取
    intent_result = IntentClassifier().classify(request.message)

    # 4. 判断是否需要追问
    if FollowUpManager().should_follow_up(intent_result, session.context):
        follow_up = FollowUpManager().generate_follow_up(intent_result, session.context, user_intent)
        return ChatResponse(session_id=session.id, message=..., follow_up=follow_up)

    # 5. 动态规划 (user_intent 影响 plan 组装)
    plan = PlanEngine().build(intent_result.intent, intent_result.slots, session.context, user_intent)
    user_intent.influence_plan(...)    # 调整计划

    # 6. 执行
    result = await PlanExecutor().execute(plan)

    # 7. 更新会话
    SessionManager().add_message(session.id, "user", request.message)
    SessionManager().add_message(session.id, "assistant", result.content, ...)
    SessionManager().update_context(session.id, {
        "user_intent_snapshot": user_intent.__dict__,
        ...
    })

    return ChatResponse(session_id=session.id, message=..., result=result)
```

### 5.4 消息交互流程

```
用户输入 "帮我选家居小电器，夏天上新，利润高的"
  → POST /api/chat/send { session_id, message }
  → 后端:
      意图 = product_selection
      槽位 = { category: "家居小电器", season: "夏季", preferences: ["高利润"] }
      槽位缺失 = [price_min, price_max]
      → FollowUpManager 生成追问
  → 返回:
      {
        "message": { "content": "好的，请问您的预算范围是？", "message_type": "slot_prompt" },
        "follow_up": { "question": "预算范围", "options": ["50-100", "100-300", "300-500", "不限"] }
      }
  → 前端渲染 SlotPromptCard，用户点击 "100-300"
  → 自动发送 "价格 100-300"
  → 后端:
      意图 = slot_reply
      合并槽位 → 槽位齐全
      → PlanEngine → PlanExecutor → 调 Agent
  → 返回:
      {
        "message": { "content": "分析完成，为您推荐以下 5 个商品...", "message_type": "text" },
        "result": { "result_type": "selection_report", "products": [...] }
      }
  → 前端渲染 text 气泡 + 更新右侧 ProductSidebar
```
